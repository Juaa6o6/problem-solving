#### math.factorial vs 반복문
- math.factorial(메모리 129272KB, 시간 356ms)
- while(메모리 2139002KB, 시간 3400ms)

1. 왜 math.factorial이 압도적으로 빠른가?

    - 파이썬의 math 모듈은 내부적으로 C 언어로 작성되어 컴파일되어 있음
    - 반복문 코드: 파이썬 인터프리터가 한 줄씩 코드를 해석하며 실행. 10만 번의 곱셈을 파이썬 엔진 위에서 돌림
    - .math.factorial: 파이썬 엔진을 거치지 않고, CPU가 이해하기 가장 쉬운 기계어에 가까운 C 코드로 즉시 연산
    - 따라서 $N$의 값이 커질수록 라이브러리를 사용하는 것이 성능 면에서 훨씬 유리
    
2. 가장 좋은 전략 
    - 학습 단계: 반복문이나 재귀로 직접 구현해 보는 것이 중요. 자료구조의 흐름과 메모리 관리(스택)를 이해해야 하기 때문
    - 실전 코딩 테스트: 문제의 의도가 '구현' 그 자체라면 직접 짜야 하지만, 팩토리얼 계산이 문제 해결의 '수단'일 뿐이라면 math.factorial을 사용하는 것이 시간 단축과 오타 방지 면에서 훨씬 유리
    - 실무 개발: 고민할 필요 없이 가장 빠르고 검증된 math 라이브러리를 사용

### 메모

#### 런타임 에러(RuntimeError) 발생 -> 재귀 깊이 제한
1. 콜 스택(Call Stack)의 개념

    프로그램에서 함수가 실행될 때, 컴퓨터는 해당 함수가 끝난 뒤 돌아갈 위치와 현재 변수 값들을 메모리의 스택(Stack)이라는 공간에 차곡차곡 저장합니다.

    재귀 함수는 자기 자신을 계속 호출하는 구조입니다. 즉, recure(n)이 끝나기 전에 recure(n-1)을 부르고, 또 그 안에서 recure(n-2)를 부릅니다. 결과적으로 맨 마지막 기저 조건(num == 0)에 도달할 때까지 앞선 함수들은 종료되지 않은 상태로 메모리 스택에 계속 쌓이게 됩니다.

2. 스택 오버플로우(Stack Overflow)

    파이썬은 메모리 과부하를 막기 위해 함수가 자기 자신을 부를 수 있는 최대 횟수(Recursion Depth)를 내부적으로 엄격하게 제한하여 허용된 메모리 용량을 초과하여 프로그램이 강제 종료되는 스택 오버플로우 현상이 발생하지 않도록 함

    - 파이썬은 기본 제한은 보통 1,000번임

3. 런타임 에러

    내부적으로 재귀 호출이 가능한 최대 횟수(기본값 보통 1,000번)를 미리 제한해 스택이 1,000번 이상 깊어지려고 하면 파이썬 인터프리터가 이를 감지하고 강제로 RecursionError(런타임 에러의 일종)를 발생시켜 프로그램을 중단시킵니다. 특히 백준 채점 서버는 이 제한이 더 엄격하게 설정되어 있는 경우가 많습니다.

<br/>

만약  입력 범위가 $0 \le n \le 100000$ 이라는 것은 출제자가 "이 문제는 재귀 함수로 풀지 말라"고 명시적으로 의도한 것

#### 반복문(Iteration)으로의 전환
- 재귀 함수가 내부적으로 의존하던 '스택 메모리'를 사용하는 대신, 결괏값을 저장할 변수 하나를 두고 반복문(for 또는 while)을 순회하며 값을 누적하여 곱하는 방식
- 반복문의 장점:함수를 메모리에 올리고 되돌아가는 복잡한 과정(Context Switching 및 Overhead)이 생략되므로 실행 속도가 압도적으로 빠름 
- 또한, 추가적인 스택 메모리를 전혀 사용하지 않으므로 $N=100,000$이 입력되어도 시스템에 무리를 주지 않음
- 시간 복잡도는 동일하게 $O(N)$이지만, 공간(메모리) 복잡도가 $O(N)$에서 $O(1)$로 획기적으로 최적화됨
